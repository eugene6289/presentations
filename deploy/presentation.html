LAYER 1  Orchestrator 統籌調度者

SKILL.md — 入口，只做偵測 + 詢問 + 分流

做什麼

  • 1 解析 git remote，識別 REPO_NAME
  • 2 AskUserQuestion：測試機 or 正式機？
  • 3 路由到對應 flow，把參數帶過去

不做什麼

✗  不執行 git 指令
✗  不決定版本號
✗  不知道 PR body 長什麼樣
✗  不處理衝突

LAYER 2  Flows

兩條端到端情境，各自完整描述一次部署的全部步驟

test-deploy.md（T-1 ～ T-9）

  • T1部署前檢查（dirty / 落後）
  • T2推送開發分支
  • T3決定版本號
  • T4合併到集合式 branch
  • T5建立並推送 Tag
  • T6觸發 GitHub Actions
  • T7切回原始分支 / stash pop
  • T8顯示部署摘要
  • T9（選填）產生 Commit Log

production-pr.md（P-0 ～ P-9）

  • P0部署前檢查（dirty）
  • P1詢問 Redmine 票務連結
  • P2確認開發分支
  • P3詢問 Release 日期與功能名稱
  • P4建立 prerelease branch
  • P5合併開發分支
  • P6選擇 Reviewer
  • P7產生 PR 說明（讀 diff）
  • P8確認並建立 PR
  • P9顯示結果，切回分支

LAYER 3  Behaviors

5 個可複用的原子行為，跨 flow 共用

handle-dirty-worktree
偵測未 commit 的變更，提供 Commit / Stash / 中止 三種選擇
測試機 正式機
handle-conflict
merge 衝突時顯示衝突清單、abort merge、清理臨時 branch
測試機 正式機
version-suggest
讀取現有 tags，推算下一版本號，提供 patch / hotfix / 自訂 三選項
測試機
reviewer-select
從 GitHub API 動態抓取 collaborators，過濾自己後讓使用者多選
正式機
commit-log
從兩個 tag 之間的 commits 產生口語化、去技術術語的主管摘要
測試機
