背景 · 這是什麼

「玩得到的廣告」是什麼？

Google 在 App 廣告類別下，允許廠商上傳一個小遊戲當作廣告素材。使用者滑手機時，不是看一張圖，而是直接玩一段 10–30 秒的小遊戲，玩完才看到「下載」按鈕。
目前的廣告

看 → 點下載

使用者看到一張圖或一段影片，覺得有興趣就點按鈕。被動觀看，互動低。
看到圖 → 點下載
這次要做的

玩 → 點下載

使用者直接在廣告位置玩一段 mini-game，體驗 App 核心玩法，再決定下載。已被驗證能大幅提升安裝意願。
看到廣告 → 玩 15-30 秒 → 點下載
Google 官方名稱：HTML5 / 可試玩廣告 (playable ad)　·　截至 2026 仍是 App 廣告活動專屬
平台規則 · Google 給的盒子

Google 給的「小盒子」有多大

這幾條是 Google 寫死的規格，沒有商量空間。它決定我們能做什麼樣的遊戲。
5MB
遊戲總大小
所有程式碼、圖片、音效、字型加起來不能超過 5 MB。這是最關鍵的限制。
Apponly
只能配 App 廣告
必須要有已上架的 App（Android / iOS）。網頁版產品不能用這個廣告類型。
CPE
玩第一下就計費
使用者第一次互動或點按時，Google 才向我們收費。不點不玩不收錢。
30天
轉換回溯期
玩完當天起 30 天內安裝 App，都算這支廣告帶來的轉換。
其他不能存進度（沒有 localStorage）
其他外部連線只能用 Google 自家函式庫白名單
其他音效要等使用者互動後才能播
可行性 · 5 MB 能做什麼

5 MB 能塞什麼樣的遊戲

5 MB 不大也不小，能做的是「微型、短時、單一玩法」的 mini-game。下面把適合 / 不適合分清楚。
✓ 適合做

短打、單一玩法、15–30 秒玩完

2D 為主，玩法越單純越好。下面這些類型都驗證過可以塞進 5 MB：
  • 點擊 / Tap 類（打地鼠、節奏點擊）
  • 消除 / Match-3 類
  • 跳躍 Runner（Tap-to-jump）
  • 瞄準射擊（拉弓、彈彈球）
  • 簡單拖放解謎
關鍵：玩 1 輪就有成就感，不需要連線、不需要存檔。
✗ 不適合做

這些直接放棄，5 MB 裝不下或不能做

不是技術問題，是平台規則本身就不允許：
  • 3D 遊戲（引擎本身就吃 1–2 MB）
  • 大量音效 / 語音的遊戲
  • 需要連線多人對戰
  • 需要存進度 / 高分榜
  • 需要超過 30 秒的關卡
如果想做的玩法落在這邊，要重新思考遊戲設計。

WHY THIS GENRE

為什麼是這類玩法最常見

Tap 打地鼠、跳躍 Runner、Match-3 這類 hyper-casual 是可試玩廣告的主流，因為：

  • 規則 1 秒看懂，不需教學
  • 玩一輪 10–30 秒就有成就感
  • 美術需求低，輕鬆壓進 5 MB
  • 玩完自然接「下載」收尾畫面

← 左邊就是手刻的範例。整段程式不到幾十 KB，剩下的 5 MB 預算都能留給美術。

建議路線

推薦的做法

兩條路：(A) 工程師手刻，彈性最高；(B) Google Web Designer，不寫程式但互動有上限。建議選 A。
建議方案

工程師手刻 HTML5 + Canvas

01
引擎成本 0 KB
不依賴任何遊戲引擎，5 MB 預算全部留給遊戲內容（圖、音、玩法）。
02
動畫用 Google 免費代管
用 Google 白名單裡的 GSAP 函式庫處理動畫，不吃我們的 5 MB 額度。
03
客製化彈性最高
完全控制玩法、視覺、節奏。不被工具元件綁住，能對齊 App 的核心體驗。
替代方案
Google Web Designer（免費官方工具）：不需要工程師，拖拉介面就能產出符合規格的 ZIP。但互動深度有限——適合做「動態廣告」、不適合做「遊戲」。如果預算極緊可以考慮。
⚠ 不推薦：Phaser / PIXI / Three.js — 主流遊戲引擎光是 minified 就要吃掉 0.5–1 MB，等於先燒掉 1/5 的預算。
實作 · 怎麼生出檔案

一個 index.html，打包成 ZIP 上傳

程式、CSS、圖片全部 inline 進同一個 index.html，再壓成 ZIP。Google 上傳要的是 ZIP，不是裸 HTML。
寫 index.html（全部 inline） → 圖片壓縮 / base64 嵌入 → 打包成 ZIP ≤ 5 MB → 上傳到 Google Ads
SINGLE FILE

全部塞進一個檔

HTML / CSS / JS 全部 inline，圖片用 base64 直接嵌入。外部檔案連結在廣告沙箱常載不出來，single-file 最保險。

IMAGES

重點是「加起來 ≤ 5 MB」

不是張數限制，是總量。圖一律壓成 WebP / 壓過的 PNG，sprite sheet 合併減少張數。base64 會膨脹約 +33%，要算進去。

建議圖片總量 ≤ 2–3 MB，留 buffer
ZIP SPEC

打包規格

ZIP ≤ 5 MB、最多 512 個檔，可放 HTML / CSS / JS / GIF / PNG / JPG / SVG（不能再包 ZIP）。每個廣告群組最多上 20 個 ZIP。

動畫用 GSAP（Google 代管，不吃額度）
技術細節：UTF-8 編碼　·　響應式全螢幕　·　音效要等使用者互動才播　·　SVG 不能用自閉合標籤
交付 · 怎麼上傳

兩條上傳路線

做好的 ZIP 有兩種產出與上傳方式。建議走 A（手刻）拿到玩法彈性，必要時用 B 的工具幫忙打包驗規格。
路線 A · 工程師手刻
建議
  1. 工程師寫 index.html（inline、圖 base64）
  2. 對規格：UTF-8、響應式、互動才播音效
  3. 壓成 ZIP（≤ 5 MB）
  4. Google Ads → App 廣告活動 → 上傳 HTML5 素材
  5. 系統自動驗規格 → 預覽 → 過審投放
玩法、視覺完全可控，能做出真正的 mini-game。缺點：規格雷區要自己顧。
路線 B · Google Web Designer
替代
  1. 用 GWD 免費官方工具拖拉做素材
  2. 選 Google Ads → Publish
  3. 自動產出符合規格的 ZIP
  4. 同樣到 Google Ads 後台上傳
不太會踩規格錯誤、不用工程師。缺點：互動深度有限，適合動態廣告、不適合真的能玩的遊戲。
建議：主力走 A 拿玩法彈性，最後若要省事可把產出丟進 GWD 走一次 publish 驗證 ZIP 合規；工程師熟的話直接手壓 ZIP 上傳也沒問題。
⚠ 硬前提：App 必須已上架（Android / iOS）。可試玩廣告只能掛在 App 廣告活動底下，純網頁產品用不了。
設計協作 · Designer 的角色

需要設計師配合的部分

設計師不是做完圖就結束，而是要全程跟工程師一起顧 5 MB 預算。下面是主要的協作項目。
01

視覺風格 & 角色 / UI 設計

定調整體美術風格，對齊 App 核心體驗；產出角色、場景、按鈕等 UI 元件。決定遊戲第一眼的吸引力。

02

Sprite Sheet 製作與合併

把動畫逐格圖合併成單張 sprite sheet，減少檔案數、方便壓縮，也讓動畫在 Canvas 上跑得更順。

03

5 MB 素材壓縮把關

與工程師一起拆每個資產的 KB 額度，圖片轉 WebP / 壓過的 PNG、去除冗餘色彩與留白。這是最吃設計師的環節。

04

結束畫面 End Card 與 CTA 視覺

玩完後的「下載」收尾畫面、CTA 按鈕視覺。直接影響轉換率，要與行銷的下載訊息對齊。

協作模式：設計師不是一次交件，而是分階段——啟動第 1 週先定風格與 KB 額度，中段交 sprite / UI，後段與工程師一起做壓縮微調，確保整包壓在 5 MB 內。
